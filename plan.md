## 🎓 Part A. 인증과제용 시스템 요구사항 정의서 (배경·전제)

> ⚠️ **이 Part A 전체는 원본 PDF (1)의 "인증과제용" 내용입니다.** 코스웍 실습의 *입력 자료*로서 여기 요약해 둡니다. 실제 실습 진행은 Part B부터입니다. (상세 원문은 [정리본](<AI기반 카페 모바일 오더 및 픽업 알림 시스템 구축_20260705_revised_convert.md>) 참조)

### A-1. 시스템 개요 (요구사항 문서 1장)
- 소상공인 카페·프랜차이즈용 **AI 기반 모바일 사전주문·결제·픽업 알림·운영 최적화** 서비스
- 연계 대상: 고객 App, 매장 직원 화면, 관리자 Web, 외부 결제, 외부 알림, AI 추론 엔진
- 처리 흐름 6단계: ① 고객/매장 채널 → ② 매장/메뉴 관리 → ③ 주문/결제 → ④ AI 추론 → ⑤ 제조/픽업알림 → ⑥ 운영·AI 대시보드

### A-2. 핵심 기능 요구사항 (요구사항 문서 2장)
- 2.1 매장·메뉴 정보 관리 / 2.2 모바일 주문·결제 / 2.3 매장 주문 처리·픽업 알림
- 2.4 AI 기반 주문 예측·운영 최적화 / 2.5 매장 운영 관리·통계 / 2.6 사용자·보안 관리(RBAC)
- **주문 상태 8종**: 주문대기 · 결제완료 · 접수완료 · 제조중 · 제조완료 · 픽업완료 · 주문취소 · 결제실패

### A-3. 시스템 품질요건 (요구사항 문서 3장) — 아키텍처 드라이버의 원천
- 3.1 성능 / 3.2 처리량·동시성 / 3.3 가용성·장애대응 / 3.4 신뢰성·데이터 일관성 / 3.5 보안·개인정보
- 3.6 AI 신뢰성·모델 관리 / 3.7 변경용이성·확장성 / 3.8 상호운용성·외부 API / 3.9 감사추적성·백업·복구 / 3.10 사용성·호환성

### A-4. 절대 놓치면 안 되는 설계 제약 (요구사항에서 반복 강조)
- **Graceful Degradation**: 외부 결제·알림·**AI 추론 엔진이 죽어도 주문·결제·제조 상태 관리는 유지** (3.3, 3.6 반복)
- **데이터 일관성**: 주문 폭주에도 중복 주문·중복 결제·주문 누락 방지 (3.4)
- **AI는 '참고 정보'**: 최종 결정은 사람. 룰 기반 대체·버전관리·롤백 가능. AI 초점은 모델 성능이 아니라 **연계·저장·예외처리·모델관리 구조** (2.4, 3.6)
- **개인정보/결제정보**: 카드번호 내부 미저장, AI 데이터 비식별화·마스킹, 전 구간 TLS (2.6, 3.5)
- **처리량 목표**: 초기 분당 10,000건 → 확장 40,000건, 수평 확장·비동기 큐 (3.2)

> 🎓 Part A는 **개인 인증과제 보고서 1~3장(Project/System Overview, Architectural Driver)의 근거**가 되는 내용이기도 하다. 코스웍에서는 이걸 "주어진 전제"로 삼아 Part B 실습을 수행한다.

## 📝 Part B. 코스웍 실습 로드맵 (우리 조가 하는 일) — 이 문서의 본론

### B-1. 배점 → 산출물 매핑 (목표 지점 먼저 보기)

| 최종 설계서 챕터 | 배점 | 이걸 채우는 코스웍 실습 | 핵심 산출물 |
|---|---|---|---|
| 1. Project Overview | 10% | (🎓 요구사항 1장 활용) | 사업 목표·배경·범위 요약 |
| 2. System Overview | 10% | 📝 OOAD 4.2 | 시스템 개념도, Actor, Use Case Diagram |
| 3. Architectural Driver | **35%** | 📝 요구공학 4.1 | Func/Non-func Req, **QA Scenario 명세**, Constraints |
| 4. Architecture Design | **45%** | 📝 아키텍처 설계 4.3 + 패턴 4.4 | Design Decision List, Design Approach + Rationale, QA Mapping |

> 정성 평가 4축: **명확성 / 정확성 / 설계 타당성 / 추적성.** 모든 산출물을 이 4개로 자가점검.

### B-2. 단계별 진행 (Step by Step)

#### Phase 0. 준비 (착수 전)
- [ ] 이해관계자(Stakeholder) 목록 확정: 고객, 매장 직원, 매장 관리자, 시스템 관리자, 외부 결제사, 외부 알림사, AI 운영자
- [ ] 요구사항 팀 전원 정독 (특히 2·3·4장)
- [ ] 산출물 저장 위치·네이밍·버전 규칙 합의
- [ ] **3인 역할 분담 확정** (아래 Part C)

#### Phase 1. 요구공학 실습 (4.1) — ★ Architectural Driver 35%의 원천
목표: 🎓 인증과제 3장에 들어갈 **QA Scenario 명세** 완성.
1. **[EX-1] 요구사항 도출**: Functional 10개 내외 + Non-functional 5개 내외
2. **[EX-2] Raw QAS 발굴 → 통합**: (Step5) 브레인스토밍 15개 내외 → (Step6) 통합·정제 8개 이상
   - **ISO/IEC 25010:2023** 품질특성으로 커버리지 점검(성능/가용성/신뢰성/보안/이식성/유지보수성 골고루)
3. **[EX-3] 우선순위화 + Refinement**: (Step7) 투표 → (Step8) 6요소 명세 `Source/Stimulus/Artifact/Environment/Response/Measure`
   > ⚠️ 여기서 **4.3의 Design Approach 4주제에 대응하는 QAS를 미리 심어라**: 멀티 OS(이식성) · 사용자 급증(확장성) · 백업·복구(가용성) · 통신 유출 차단(보안). 여기에 Graceful Degradation·데이터 일관성 QAS도 포함.

#### Phase 2. OOAD 실습 (4.2) — System Overview(2장) 재료
1. **[EX-1] Actor 도출** + Description
2. **[EX-2] Use Case Model** → **Use Case Diagram**
3. **[EX-3] UC 명세서**: UC 2~3개 선택, Basic Flow 중심 (optional: Sequence Diagram)
   - 선택 팁: 핵심 흐름(주문 생성→결제→접수→제조완료→픽업알림)을 고를 것

#### Phase 3. 아키텍처 설계 실습 (4.3) — ★ Architecture Design 45%
1. **[EX-1] Design Decision List**: 요구사항 중요도 평가(QA는 4.1 [EX-3] 결과 반영) → 드라이버 식별(ADD Step3) → 추상화 → DDL 작성 → ISO25010 리뷰 → QA Mapping → 확정 → 발표
2. **[EX-2~5] Design Approach 4종** — 각각 `Design Goal → Design Approach → Design Decision & Rationale → 발표 → Review`:

   | 실습 | 주제 | 대응 품질속성 | Tactic/Pattern 힌트 |
   |---|---|---|---|
   | EX-2 | 멀티 OS 대응 | 이식성·사용성 | 추상화 계층, Adapter, 반응형/크로스플랫폼 |
   | EX-3 | 급격한 사용자 증가 | 확장성·성능 | 수평 확장, 비동기 큐, 로드밸런싱, 캐시 |
   | EX-4 | 데이터 백업·복구 | 가용성·신뢰성 | 주기적 백업, 이력/감사 로그, 복구 절차 |
   | EX-5 | 통신 데이터 유출 차단 | 보안 | TLS/HTTPS, 민감정보 미저장, 마스킹, RBAC |

#### Phase 4. Design Patterns 실습 (4.4) — 설계 타당성 심화
1. [ ] 변경 가능성 높은 기능 1개 선택 (예: 결제수단 추가, 외부 결제사 변경, 알림 채널 추가, 주문상태 추가)
2. [ ] Design Approach 대안 2~3개 (직접구현 / 단일 GoF / 복합·구조분리)
3. [ ] 비교표: 적용 패턴 · 장점 · 단점 · 변경 영향 · 관련 QA
4. [ ] 최종 선택 + 선택 이유·trade-off / (필요 시) Class Diagram
   - 후보 패턴: Strategy, Adapter, Facade, State, Observer, Command, Builder, Decorator, Factory Method

#### Phase 5. (개인) 🎓 인증과제 보고서로 통합
- [ ] 📝 Phase 1~4 산출물을 B-1 배점표에 맞춰 4개 챕터로 재배치
- [ ] **추적성 사슬 최종 점검** + 4개 정성 기준 자가 리뷰

## 📝 Part C. 3인 팀 실행 플랜 (우리 조 3명)

### C-1. 기본 전략: "1인 1 품질 드라이버" 엔드투엔드 오너십
실습이 요구공학→설계로 이어지므로, **각자 품질 도메인 하나를 맡아 QAS부터 Design Approach까지 끝까지 책임**진다. 이러면 (1) 추적성이 사람 단위로 보장되고 (2) 병렬 작업이 가능하다.

| 팀원 | 담당 품질 드라이버(엔드투엔드) | 맡는 Design Approach(4.3) | 겸임 역할 |
|---|---|---|---|
| **팀원 A** | 성능·확장성 (처리량, 피크 대응) | **EX-3** 급격한 사용자 증가 | 🔧 **요구공학(4.1) 리드** — QAS 취합·정제 총괄 |
| **팀원 B** | 가용성·신뢰성 (Graceful Degradation, 데이터 일관성) | **EX-4** 데이터 백업·복구 | 🔧 **OOAD(4.2) 리드** — Actor/UC Diagram 총괄 |
| **팀원 C** | 보안·이식성 (개인정보, 멀티 OS) | **EX-5** 통신 유출 차단 + **EX-2** 멀티 OS | 🔧 **설계(4.3) DDL 취합 + 문서·발표 통합** |

> EX-2(멀티 OS)는 4개 중 상대적으로 가벼워 보안 담당(C)이 겸한다. 부담되면 **A/B가 각자 시간 여유 시 EX-2를 분담**해도 됨.

### C-2. Phase별 3인 분담표 (누가 리드 / 병렬 여부)

| Phase | 팀원 A | 팀원 B | 팀원 C | 병렬 가능? |
|---|---|---|---|---|
| **0. 준비** | 요구사항 2·3장 정독 | 요구사항 2·3장 정독 | 요구사항 4장 정독 + 문서 세팅 | ✅ 동시 |
| **1. 요구공학** | 🔧리드: Func/Non-func 취합, QAS 통합 | 가용/신뢰 QAS 발굴 | 보안/이식 QAS 발굴 | ✅ 각자 도메인 QAS 병렬 → A가 통합 |
| **2. OOAD** | 주문/결제 UC 상세 | 🔧리드: Actor·UC Diagram, 제조/알림 UC | 보안/권한(RBAC) UC | ✅ UC별 병렬 → B가 통합 |
| **3. 설계 DDL** | 성능/확장 DD | 가용/신뢰 DD | 🔧리드: DDL 취합·ISO25010 리뷰·QA Mapping | ✅ DD 병렬 → C가 통합 |
| **3. Approach** | **EX-3** | **EX-4** | **EX-5 (+EX-2)** | ✅ 완전 병렬 |
| **4. 패턴** | 대안 비교 작성 지원 | 대안 비교 작성 지원 | 🔧리드: 기능 선정·비교표·Class Diagram | 🔶 1개 기능 공동 |
| **5. 통합(개인)** | 본인 인증과제 보고서 | 본인 인증과제 보고서 | 본인 인증과제 보고서 | ✅ 각자 |

> 🔧 = 해당 Phase의 리드(취합·발표 담당). 리드가 세 Phase에 고르게 분산되도록 A→B→C 순으로 배치.

### C-3. 협업 규칙 (권장)
- **매 Phase 종료 시 30분 싱크**: 각자 산출물 공유 → 추적성 사슬(Goal→…→Rationale) 끊긴 곳 점검
- **QAS·DDL은 한 파일에 통합 관리** (담당자별 섹션). 3명이 각자 문서를 따로 만들면 추적성이 깨진다
- **발표는 Phase 리드가 주도**하되, Q&A는 담당 도메인 오너가 답변
- **네이밍**: `{Phase번호}_{산출물}_{담당자이니셜}.md` 식으로 충돌 방지
