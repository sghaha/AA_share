# Phase 1. 요구공학 실습 (원본 4.1) — ★ 점수의 핵심

> [← Phase 0 준비](00_준비.md) · [인덱스](README.md) · [다음: Phase 2 OOAD →](02_OOAD.md)

## 🧭 이 단계가 의미하는 것
"**무엇을(기능) · 얼마나 잘(품질)** 만들지"를 문장으로 확정하는 단계입니다. 특히 **품질(성능·가용성·보안 등)을 애매한 형용사가 아니라 측정 가능한 시나리오(QAS)로 못박는 것**이 이 실습의 본질입니다.

왜 제일 중요하냐 — 배점에서 **Architectural Driver가 35%**이고, 그 실체가 바로 여기서 나오는 **QA Scenario**이기 때문입니다. 그리고 이 QAS가 3단계(설계)에서 "왜 이렇게 설계했나"의 근거가 됩니다. 즉 **여기가 부실하면 설계 45%까지 흔들립니다.**

> QAW(Quality Attribute Workshop) = 이해관계자가 모여 품질 시나리오를 발굴·우선순위화·정제하는 워크숍 기법. 이 실습은 그 축약판(Mini-QAW)입니다.

## 🎯 목표(Output)
- Functional Requirement 10개 내외 / Non-functional Requirement 5개 내외
- Raw QAS 15개 → 통합 QAS 8개 이상
- 우선순위화된 QA 목록
- **QA Scenario 명세** (상위 시나리오를 6요소로 완성)

## 📐 인증과제 설계서 어디에 들어가나
> 설계서 **3장 Architectural Driver → 3.2 Quality Attribute Scenario** (3.2.1 QA Scenario List, 3.2.2~ QA-01/02 Scenario). 제약사항은 3.3 Constraint.

## 👥 담당(3인) — 도메인별 병렬
- **팀원 A (리드)**: 성능·확장성 QAS + 전체 취합·정제
- **팀원 B**: 가용성·신뢰성 QAS (Graceful Degradation, 데이터 일관성)
- **팀원 C**: 보안·이식성 QAS
> 각자 자기 도메인 QAS를 병렬로 발굴 → A가 한 파일로 통합.

## ✅ 세부 진행 아이템

### [EX-1] 요구사항 도출
1. **기능 요구사항 10개 내외** — 요구사항 2장(6개 기능군)에서 상위 기능을 뽑아 "고객은 ~할 수 있어야 한다" 형태로.
2. **비기능 요구사항 5개 내외** — 요구사항 3장(품질요건 10절)에서 대표 품질 요구를 뽑는다.

### [EX-2] Raw QAS 발굴 → 통합
3. **(QAW Step5) 브레인스토밍** — 이해관계자 관점에서 QAS **15개 내외**. "언제(자극) → 무엇이(대상) → 어떻게 반응(측정)" 을 거칠게라도.
4. **커버리지 점검** — [ISO/IEC 25010:2023](../../../notes/03_SoftwareArchitectureFramework소개_summary.md) 품질특성(성능효율성·신뢰성·보안·호환성·유지보수성·이식성·사용성·기능적합성)으로 빠진 영역이 없는지 체크.
5. **(QAW Step6) 통합·정제** — 중복 병합, 표현 다듬어 **8개 이상**으로.

### [EX-3] 우선순위화 + Refinement
6. **(QAW Step7) 우선순위 투표** — 이해관계자 중요도 기준으로 투표 → Prioritized QA List.
7. **(QAW Step8) QAS 6요소 명세** — 상위 시나리오를 아래 6칸으로 완성:

| 요소 | 뜻 | 예 (가용성) |
|---|---|---|
| **Source** | 자극의 출처 | 외부 AI 추론 엔진 |
| **Stimulus** | 발생한 사건 | 추론 엔진 무응답(장애) |
| **Artifact** | 대상 | 주문 처리 서비스 |
| **Environment** | 상황 | 피크 시간대 정상 운영 중 |
| **Response** | 기대 반응 | 기본 평균 제조시간으로 대체 안내, 주문 기능 정상 유지 |
| **Measure** | 측정 기준 | 대체 응답 5초 이내, 주문 성공률 99% 유지 |

> ⚠️ **여기서 반드시 심을 QAS** — 3단계의 Design Approach 4주제와 1:1로 이어진다:
> - 멀티 OS 대응(이식성) · 급격한 사용자 증가(확장성·성능) · 데이터 백업·복구(가용성·신뢰성) · 통신 데이터 유출 차단(보안)
> - 여기에 이 시스템의 **최우선 제약** QAS도: ① 외부 결제/알림/AI 장애 시 핵심 주문 유지(Graceful Degradation) ② 주문 폭주 시 중복결제·누락 방지(데이터 일관성)

## 💡 작성 팁 · 자주 하는 실수
- **Measure(측정)가 없으면 QAS가 아니다.** "빠르게 응답"(X) → "평균 3초 이내"(O). 숫자는 요구사항 3장에 이미 많이 있으니 그대로 활용.
- QAS는 **한 시나리오 = 한 품질속성**. 성능과 보안을 한 시나리오에 섞지 말 것.
- FR 개수를 억지로 맞추지 말고 **아키텍처에 영향 주는 것** 위주로. 사소한 CRUD는 뺀다.
- 우선순위 투표 결과·근거를 기록해 두면 3단계 "요구사항 중요도 평가"에 그대로 재사용된다.
