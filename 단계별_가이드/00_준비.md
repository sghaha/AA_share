# Phase 0. 준비 — 판 깔기 + Overview 기초

> [← 인덱스](README.md) · [다음: Phase 1 요구공학 →](01_요구공학.md)

## 🧭 이 단계가 의미하는 것
본격 실습(요구공학~설계) 전에 **모두가 같은 그림을 보게 맞추는 단계**입니다. 요구사항을 정독하고, "이 시스템을 둘러싼 사람·조직(이해관계자)이 누구이고, 그들이 원하는 게 뭔지(비즈니스 목표)"를 정리합니다.

왜 중요하냐면 — 평가의 축인 **추적성**의 *맨 앞 고리*가 여기서 만들어지기 때문입니다:
```
[여기서 시작] Business Goal → System Feature → Architectural Driver → Design Decision → ...
```
시작점(이해관계자·비즈니스 목표)이 부실하면 뒤 단계 전부가 근거 없이 떠 있게 됩니다.

## 🎯 목표(Output)
- 이해관계자(Stakeholder) 목록
- 비즈니스 목표(Business Goal) 목록
- 시스템 기능(System Feature) 초안 목록
- (그림) Business Context Diagram / System Context Diagram 초안
- 팀 규칙(산출물 저장·네이밍·역할) 확정

## 📐 인증과제 설계서 어디에 들어가나
> 이 단계 산출물은 설계서 **1장 Project Overview + 2장 System Overview**의 재료입니다.

| 산출물 | 설계서 위치 |
|---|---|
| 이해관계자 목록 | 1.3 Stakeholder List |
| 비즈니스 목표 | 1.4 Business Goal List |
| Business Context Diagram | 1.2 |
| System Context Diagram | 2.1 |
| External Entity / Interface | 2.2 / 2.3 |
| System Feature List | 2.4 |
| 가정(Assumptions) | 2.5 |

## 👥 담당(3인)
- **전원**: 요구사항 정독(팀원 C는 4장 실습 안내 집중)
- **팀원 C**: 문서 저장소·템플릿 세팅(설계서 양식의 표를 미리 빈칸으로 복사)

## ✅ 세부 진행 아이템
1. **요구사항 통독** — [원문 정리본](<../AI기반 카페 모바일 오더 및 픽업 알림 시스템 구축_20260705_revised_convert.md>) 1~4장. 특히 1.1(구축 목적)·1.3(개념도)에서 이해관계자·목표 힌트를 뽑는다.
2. **이해관계자 식별** — 고객 / 매장 직원 / 매장 관리자 / 시스템 관리자 / 외부 결제사 / 외부 알림사 / AI 운영자. 각자 *영향력(상·중·하)·효과성(상·중·하)* 을 붙인다. (설계서 1.3 표 형식)
3. **비즈니스 목표 도출** — 이해관계자별로 "이 시스템으로 무엇을 이루려 하는가". 예: 매장주 = 대기시간↓·주문누락↓, 고객 = 줄 안 서고 픽업. 각 목표에 **BG-01…** ID와 중요도를 부여.
4. **시스템 기능(Feature) 초안** — 비즈니스 목표를 이루는 기능을 **SF-01…** 로 나열하고 관련 BG ID를 연결(2장 6개 기능군이 좋은 출발점).
5. **Context Diagram 초안** — 시스템을 가운데 두고 외부 엔티티(결제·알림·AI·지도 등)와 정보 흐름을 화살표로. (도구: draw.io / Mermaid / PPT)
6. **가정(Assumptions) 정리** — 개발 전제(예: "외부 결제사는 표준 REST API 제공", "매장당 태블릿 1대 이상")를 A-01… 로.
7. **팀 규칙 확정** — 산출물 저장 위치, 파일 네이밍(`{Phase}_{산출물}_{이니셜}`), [3인 역할](../과제_진행_가이드.md#part-c-3인-팀-실행-플랜-우리-조-3명) 재확인.

## 💡 작성 팁 · 자주 하는 실수
- **이해관계자 = 액터가 아니다.** 이해관계자는 "관심을 가진 사람/조직"(투자자·규제기관 포함 가능), 액터(2단계)는 "시스템과 직접 상호작용하는 대상". 겹치지만 다르다.
- Business Goal은 **측정 가능하게**. "편해진다"(X) → "픽업까지 평균 대기 5분 이내"(O). 뒤의 QA 시나리오와 연결된다.
- Context Diagram의 외부 엔티티는 나중에 **Use Case의 액터와 반드시 일치**해야 한다(설계서 2.1 점검기준). 미리 맞춰두면 2단계가 쉬워진다.
