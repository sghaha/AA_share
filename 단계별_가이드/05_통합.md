# Phase 5. 통합 — 흩어진 산출물을 하나의 보고서로

> [← Phase 4 디자인 패턴](04_디자인패턴.md) · [인덱스](README.md)

## 🧭 이 단계가 의미하는 것
Phase 0~4에서 각자·각 단계별로 만든 산출물을 **하나의 인증과제 설계서로 꿰는** 단계입니다. (이 단계는 조별 코스웍이 아니라 **개인 인증과제**로 넘어가는 지점 — 각자 자기 보고서를 완성합니다.)

의미상 이건 "조립과 검증"입니다. 개별 산출물이 아무리 좋아도 **추적성 사슬이 끊기면** 설계서로서 낙제입니다. 이 단계의 진짜 일은 새 내용을 쓰는 게 아니라, **연결을 확인하고 빈틈을 메우는 것**입니다.

## 🎯 목표(Output)
- 완성된 소프트웨어 아키텍처 설계서 (설계서 양식 4개 챕터)
- 추적성 점검 통과
- 4개 정성 기준(명확성·정확성·설계 타당성·추적성) 자가 리뷰 통과

## 📐 인증과제 설계서 전체 매핑
| 설계서 챕터 | 배점 | 채우는 산출물 | 나온 단계 |
|---|---|---|---|
| 1. Project Overview | 10% | Charter, Stakeholder, Business Goal | Phase 0 |
| 2. System Overview | 10% | Context Diagram, External Entity/Interface, System Feature, Assumptions | Phase 0 |
| 3. Architectural Driver | **35%** | Use Case Model + QA Scenario + Constraints | Phase 1·2 |
| 4. Architecture Design | **45%** | Design Decision List + Design Approach + (패턴 상세) | Phase 3·4 |

## ✅ 세부 진행 아이템
1. **산출물 재배치** — 위 표대로 각 산출물을 설계서 양식의 해당 챕터·표에 옮긴다. (양식의 표 형식 그대로 만들어 뒀다면 복사만 하면 됨.)
2. **누락 챕터 보완** — 코스웍 실습에서 안 다룬 부분(예: Project Charter, Constraints, Assumptions)을 요구사항 기반으로 채운다.
3. **추적성 사슬 점검** — 아래를 한 줄씩 따라가며 끊긴 고리를 찾는다:
   ```
   Business Goal → System Feature → (Use Case / QA Scenario) → Design Decision → Design Approach → Rationale
   ```
   - 각 Design Decision → 근거 QAS가 있는가?
   - 각 QAS → 관련 System Feature/Business Goal로 거슬러 올라가는가?
   - ID(BG/SF/UC/QA/DD)로 상호 참조가 걸려 있는가?
4. **정성 4기준 자가 리뷰** — [배점 기준](../../../평가기준_설계산출물.md) 표를 체크리스트로:
   - 명확성(용어·기술이 명확한가) / 정확성(UC·QAS·설계 표현이 정확한가) / 설계 타당성(결정 논리·대안 다양성·Rationale 충실) / 추적성(위 사슬 연결)
5. **양식 점검기준 대조** — [설계서 양식](../../../certification/SW아키텍처인증과제_설계서양식_작성가이드_20260625_convert.md)의 각 절 "점검기준"을 하나씩 대조.
6. **최종 검토·제출 형식 정리** — 표/그림 번호, 캡션, 목차 정리.

## 💡 작성 팁 · 자주 하는 실수
- **추적성은 ID로 증명한다.** 문장으로 "연결되어 있다"가 아니라, DD-03의 근거에 `QA-02` 라고 실제 ID를 박아라.
- 배점이 큰 3장(35%)·4장(45%)에 분량·완성도를 몰아준다. Overview(10+10)는 간결하게.
- 코스웍은 조별이지만 **인증과제는 개인 제출** — 공통 산출물을 각자 자기 언어로 재정리해 이해도를 보여줄 것.
- 마감 전 팀 상호 리뷰 1회: 서로의 추적성 사슬을 교차 점검하면 빈틈이 잘 보인다.
