# Phase 4. Design Patterns 실습 (원본 4.4) — 결정을 코드 구조로

> [← Phase 3 아키텍처 설계](03_아키텍처설계.md) · [인덱스](README.md) · [다음: Phase 5 통합 →](05_통합.md)

## 🧭 이 단계가 의미하는 것
3단계의 굵직한 설계 결정을 **"변경에 강한 코드 구조"** 수준까지 내려서 정교화하는 단계입니다. **변경 가능성이 높은 기능 하나**를 골라, 그것을 어떻게 설계하면 나중에 바꾸기 쉬운지 **GoF 디자인 패턴 대안들을 비교**합니다.

의미상 이건 "설계 타당성"을 증명하는 단계입니다. 아키텍트의 실력은 *변화를 어디에 가둘지*를 아는 데서 드러납니다. 카페 시스템에서 가장 잘 바뀌는 것 — 결제수단, 알림 채널, 주문 상태 — 을 패턴으로 격리해 두면 확장성/변경용이성(요구사항 3.7)을 구조로 증명하게 됩니다.

> 이 실습의 중심은 Design Decision List 재도출이 아니라 **패턴을 활용한 Design Approach 대안 비교**입니다.

## 🎯 목표(Output)
- 변경 기능 1개 + 변경 시나리오
- Design Approach 대안 2~3개 비교표
- 최종 선택 + 선택 이유·trade-off
- (필요 시) 주요 클래스 Class Diagram

## 📐 인증과제 설계서 어디에 들어가나
> 설계서 **4장 Architecture Design**을 보강하는 상세 설계 근거. 3단계의 DD 중 "변경용이성" 관련 결정을 패턴으로 뒷받침.

## 👥 담당(3인)
- **팀원 C (리드)**: 기능 선정·비교표·Class Diagram 총괄
- **팀원 A·B**: 대안 분석·장단점 검토 지원
> 1개 기능을 팀이 함께 파고든다.

## ✅ 세부 진행 아이템
1. **변경 가능성 높은 기능 1개 선택** — 예: 결제수단 추가 / 외부 결제사 변경 / 알림 채널 추가 / 주문 상태 추가 / 메뉴 옵션 추가. (Phase 2에서 고른 UC와 연결하면 좋음)
2. **변경 시나리오 정의** — "지금은 카드결제만. 6개월 뒤 간편결제·포인트결제 추가 예정" 처럼 구체적으로.
3. **Design Approach 대안 2~3개 작성**:
   - 대안 1: 단순 직접 구현(if-else 분기)
   - 대안 2: 단일 GoF 패턴 적용
   - 대안 3: 복합 패턴 또는 구조 분리
4. **비교표 작성** — 각 대안에 대해: 적용 패턴 · 장점 · 단점 · 변경 영향 · 관련 Quality Attribute.
   > 후보 패턴: **Strategy**(교체 가능한 알고리즘 - 결제수단), **Adapter**(외부 시스템 연동 - 결제사/알림사), **Observer**(이벤트 알림 - 픽업 알림), **State**(주문 상태 전이), **Facade**(복잡한 하위 시스템 단순화), Command, Builder, Decorator, Factory Method.
5. **최종 선택 + 근거** — 왜 이 대안인지, 무엇을 얻고 무엇을 포기하는지(trade-off).
6. **(필요 시) Class Diagram** — 주요 클래스명·관계 중심의 간단한 다이어그램.

### 이 시스템에 어울리는 예시 매핑(참고)
| 변경 기능 | 잘 맞는 패턴 | 이유 |
|---|---|---|
| 결제수단 추가 | Strategy | 결제 알고리즘을 런타임 교체 |
| 외부 결제사/알림사 교체 | Adapter | 외부 API 차이를 내부 인터페이스로 흡수 |
| 픽업 알림(Push/SMS/메신저) | Observer + Strategy | 이벤트 구독 + 채널별 발송 전략 |
| 주문 상태 전이 | State | 8종 상태 전이 규칙을 상태 객체로 |

## 💡 작성 팁 · 자주 하는 실수
- **대안 1(직접 구현)을 반드시 포함하라.** "패턴을 안 썼을 때의 고통"을 보여줘야 패턴의 가치가 드러난다.
- 패턴을 위한 패턴 금지. **변경 시나리오가 실제로 요구하는** 유연성에만 패턴을 쓴다(과설계는 감점).
- 비교표에 **관련 Quality Attribute를 반드시** 붙여 Phase 1 QAS와 추적성을 잇는다.
- Class Diagram은 화려할 필요 없다. 핵심 인터페이스와 구현 클래스 관계만 명확히.
